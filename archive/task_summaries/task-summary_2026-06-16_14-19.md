# 经验教训：Dalin Soma 论文PDF生成与发布策略（2026-06-15 ~ 2026-06-16）

## 一、PDF 生成：从 fpdf2 到 Playwright 的完整历程

### 核心问题
用户要求将 Dalin Soma 五篇学术论文（中英文双语）生成 PDF，但中英混排 + CJK 字符是巨大的技术障碍。

### 尝试的三条路径

#### ❌ 路径1: fpdf2 直�C生成
- **问题**：fpdf2 默认字体（Courier）不支持 CJK，出现 `UnicodeEncodeError`
- **尝试修复**：换 DejaVu Sans → 仍不支持 CJK，出现 `` 黑框
- **结论**：fpdf2 没有字体嵌入功能，对中英文混排内容不可用
- **教训**：**纯 Python 的 PDF 库如果不自带字体管理，基本无法处理多语言混合排版**

#### ❌ 路径2: wkhtmltopdf / pandoc
- **问题**：系统未安装这些工具
- **教训**：**依赖外部工具链前应先检查依赖可用性，避免空跑**

#### ✅ 路径3: Playwright HTML → PDF
- **思路**：Markdown → Python 解析提取英文 → 生成干净英文 HTML → Playwright 截图转 PDF
- **优势**：Playwright 使用系统字体（如 San Francisco），Unicode 支持完美
- **最终结果**：成功生成 5 个独立 PDF + 1 个合并 PDF，~99.9% 英文内容，剩余 ~20 个中文字符（算法伪代码标签、表格标题等边缘案例）

### 关键技术细节

1. **抽象提取陷阱**：`> **Abstract:**` 同时包含 `**` 和 `:`，会被前面的 metadata 检查提前消耗。修复：把 abstract 检查移到 metadata 检查之前。
2. **笔记泄漏到抽象**：后续 `> **重要声明**` 等笔记行以 `> ` 开头，被错误追加到 abstract_text。修复：增加 `not s.startswith('> **')` 条件。
3. **CJK 过滤阈值**：从 `alpha_count < 3` 改为 `cjk_count > (cjk_count + alpha_count) * 0.6`，用 CJK 比例判断段落语言倾向。
4. **代码块 CJK 剥离**：算法伪代码中的 "输入:" "输出:" 等中文标签，在代码块渲染时通过 `re.sub(r'[\u4e00-\u9fff]', '', s)` 剥离。
5. **数学公式**：HTML→PDF 无法自动渲染 LaTeX，公式以原始字符串显示（如 `$$S_i(x,t) = ...$$`）。接受为草案阶段的合理妥协。

### 经验总结

| 问题 | 教训 |
|------|------|
| fpdf2 CJK | 纯 Python PDF 库不适合多语言混合，优先用浏览器渲染引擎 |
| 解析顺序 | 结构化标记（如 abstract blockquote）必须在通用规则（如 metadata）之前检查 |
| 阈值策略 | 用比例而非绝对计数来判断语言倾向更鲁棒 |
| 外部依赖 | 先检查依赖可用性再尝试，避免无效循环 |
| 数学公式 | HTML→PDF 不渲染 LaTeX，如需专业公式渲染需 MathJax/KaTeX |

## 二、arXiv 投稿受阻 → 转向中国技术社区

### 遇到的问题
1. **背书门槛**：arXiv cs.LG 分类需要已有背书者推荐，独立研究者无法获得
2. **推荐码**：生成唯一验证码 ZFI67X 但无法找到符合资格的推荐人
3. **邮件求助**：尝试联系 help@arxiv.org 但操作困难

### 策略调整
- 放弃 arXiv（独立研究者的系统性障碍）
- 转向 **掘金（Juejin）** 发布技术文章
- 保持 MIT 开源许可
- 先发布技术报告 → 积累社区反馈 → 未来再冲击 ICLR/NeurIPS

### 经验总结
- arXiv 的背书系统对独立研究者极不友好，不是技术问题而是制度障碍
- 对于非学术机构研究者，中国技术社区（掘金、知乎）是更现实的第一发布渠道
- 可以先发布技术报告版本，积累引用和社区验证后再投顶会
