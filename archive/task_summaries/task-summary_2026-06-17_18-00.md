# Task Summary: Deep Review of Dalin Soma Integration

## Objective
Comprehensive review of the Dalin Soma (SFA) integration in llama.cpp from three perspectives: maintainer (code quality/architecture compliance), user (practical usability), and academic integrity (paper claims vs actual code).

## Key Findings

### What Works Well ✅
- Architecture registration complete (LLM_ARCH_DALIN_SOMA, "dalin-soma" string, factory method)
- Model-level integration (not custom ggml operator) — correct decision per maintainer feedback
- Dual-channel attention graph builder: near-field (FlashAttention) + far-field (EMA field state)
- Full compilation: 100% targets, 0 errors, 0 warnings
- State write/read framework with magic number validation

### Critical Bugs (P0) 🔴
1. **field_state sync mechanism is fragile**: Uses tensor name matching ("field_ema") which is unreliable; only handles single sequence; no cross-device memcpy protection
2. **llama_kv_cache_soma constructor bug**: `n_sfa_layers = hparams.n_swa` (window size, e.g., 64) should be `hparams.n_layer()` (e.g., 32)
3. **seq_cp/seq_rm index comparisons swapped**: Comparing seq_id against layer count instead of sequence count
4. **Redundant field_state management**: Both model tensor AND KV cache layer manage field_state, but they're never synchronized

### Major Concerns (P1) ⚠️
5. **K_mean calculation**: Works for decode (n_seq_tokens=1) but wrong for batch inference
6. **Parameter hijacking**: `hparams.f_attn_value_scale` and `hparams.f_attention_scale` overwritten with SFA params
7. **Code duplication**: SFA and non-SFA branches have identical build_qkv + build_attn chains

### Academic Integrity Issues ❌
- **"8.1KB extra params"**: Wrong. Actual field_state ≈ 16MB + LingYa P ≈ 4MB = ~20MB for 7B
- **"4.16× decode speedup"**: Theoretical target only. Current implementation is SLOWER than standard attention
- **"Cosine Similarity > 0.9999999"**: No verification code exists
- **"248× KV compression"**: Only theoretical; ISWA is sliding window, not true ring buffer compression

## Conclusions
- The integration is a functional skeleton but has critical bugs that cause silent failures in multi-sequence scenarios
- Paper claims need significant revision to distinguish theoretical targets from measured results
- Priority: fix field_state sync → fix KV cache bugs → run benchmarks → revise paper
