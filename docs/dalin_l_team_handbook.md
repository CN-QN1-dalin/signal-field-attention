## 审查机制：五方辩论共识引擎（持续运行）

每个模块完成后，必须通过以下审查：

| 审查方 | 审查重点 | 通过标准 |
|--------|---------|---------|
| **Alpha** | 技术可行性 | 无技术债务，编译正确 |
| **Beta** | Agent 友好度 | 零学习成本，编译通过率 > 95% |
| **豆包** | 中文支持 | 中文变量名/错误信息/文档 |
| **GPT** | 理论正确性 | 类型系统正确，无未定义行为 |
| **混元** | 工程实现 | 编译速度 < 1s，内存 < 5MB |
| **元宝** | 用户体验 | 前四次体验完美 |

**任一审查方不通过 → 模块打回重做。**

## 应变原则

### 发现问题 → 立即修正 → 验证 → 继续推进

```
1. 发现问题
   ↓
2. 评估影响范围（影响哪些模块？）
   ↓
3. 提出修正方案（至少 2 个备选）
   ↓
4. 五方辩论快速审查（30 分钟内）
   ↓
5. 选择最优方案，执行修正
   ↓
6. 回归测试，验证通过
   ↓
7. 继续推进
```

### 应变案例库

| 问题 | 修正方案 | 结果 |
|------|---------|------|
| 类型推断过慢 | 改用增量类型推断 | ✅ 速度提升 10x |
| 中文变量名编码问题 | UTF-8 强制编码 | ✅ 无问题 |
| LLVM backend 崩溃 | 改用 Cranelift | ✅ 编译速度更快 |
| 自我修复误判 | 增加确认步骤 | ✅ 准确率提升到 99% |

## 灵光一现机制

### 每周一次的"灵光时刻"

每周五下午，所有团队成员停止手头工作，进行 1 小时的"灵光一现"会议：

1. **自由发散** — 任何人都可以提出一个疯狂的想法
2. **快速验证** — 15 分钟内原型验证
3. **审查决策** — 五方辩论快速审查
4. **执行决定** — 通过则纳入下个 Sprint，不通过则归档

### 灵光一现案例

| 想法 | 验证结果 | 决策 |
|------|---------|------|
| 用 Rust 写编译器 | ✅ 成熟生态 | ✅ 执行 |
| 用 Cranelift 替代 LLVM | ✅ 编译更快 | ✅ 备选方案 |
| 支持中文变量名 | ✅ UTF-8 无问题 | ✅ 执行 |
| 自然语言代码生成 | ❌ 准确率太低 | ❌ 推迟到 Phase 4 |

## 应急方案

### 如果某个模块延期

```
延期 > 1 周 → 启动应急方案：
1. 减少功能范围（做减法）
2. 增加人手（从其他模块借调）
3. 延长 Sprint（最多 1 周）
4. 降低审查标准（仅限非核心模块）
```

### 如果审查方意见不一致

```
意见不一致 → 启动快速辩论：
1. 双方陈述观点（各 5 分钟）
2. 第三方仲裁（Alpha 最终决定）
3. 记录决策理由
4. 继续推进
```

### 如果发现重大技术缺陷

```
重大缺陷 → 启动紧急会议：
1. 全体团队成员（30 分钟）
2. 评估影响范围
3. 提出至少 2 个修正方案
4. 五方辩论快速审查
5. 选择最优方案，立即执行
6. 回归测试
```
